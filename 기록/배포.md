🚀 Spring Boot + React + MariaDB 배포 가이드 (GCP & Docker)
1. 배포(Deployment)란 무엇인가?
내 컴퓨터(Local)에서 개발한 프로그램을 **24시간 꺼지지 않는 외부 서버(Cloud)**에 올려서, 전 세계 사람들이 주소(Domain/IP)를 통해 접속할 수 있게 만드는 과정입니다.

2. 왜 GCP와 Docker를 사용하는가?
GCP (Google Cloud Platform): 구글의 인프라를 빌려 쓰기 때문에 서버 관리가 안정적이며, 무료 크레딧을 통해 비용 부담 없이 시작할 수 있습니다.

Docker: "어느 환경에서나 동일하게 실행되는 박스"입니다. 내 컴퓨터에서 잘 되던 게 서버에서 안 되는 문제를 원천 차단하며, 명령어 한두 줄로 복잡한 DB나 서버 설정을 끝낼 수 있습니다.

장점: 확장성이 좋고, 나중에 배포 자동화(CI/CD)를 구축하기에 가장 최적화된 조합입니다.

3. 전체 배포 프로세스 요약
GCP 설정: VM 인스턴스 생성 및 방화벽(80, 8080, 5173, 3306) 개방.

도커 이미지 빌드: 내 컴퓨터의 코드를 Docker 이미지로 구움.

이미지 푸시: 구글 저장소(Artifact Registry)에 이미지 업로드.

서버에서 실행: GCP SSH 접속 후 이미지를 내려받아(Pull) 컨테이너 실행.

도메인 연결: IP 주소를 '내도메인 한국' 등을 통해 문자 주소로 연결.

4. 핵심 명령어 모음 (Cheat Sheet)
Step 1. 내 컴퓨터에서 박스 만들기 (Build & Push)
수정한 코드를 클라우드로 보내기 위해 '도커 이미지'라는 압축 파일로 만드는 과정입니다.

```Bash
# 1. 백엔드 폴더로 이동 후 이미지 생성
# -t: 이미지에 이름을 붙임 (이름:태그 형식, 태그 생략 시 latest)
# .: 현재 폴더의 Dockerfile을 읽으라는 뜻
docker build -t diary-backend .

# 2. 구글 창고(Artifact Registry) 주소에 맞게 이름표 다시 달기
# 형식: docker tag [기존이름] [구글창고주소]/[이미지명]
docker tag diary-backend asia-northeast3-docker.pkg.dev/haroo-799a8/diary-repo/backend-app

# 3. 구글 창고로 업로드 (인터넷 속도에 따라 시간이 걸림)
docker push asia-northeast3-docker.pkg.dev/haroo-799a8/diary-repo/backend-app
```
⚠️ 주의: docker build 전, 도커 데스크탑이 켜져 있는지 꼭 확인하세요!

Step 2. GCP 서버 청소 및 준비 (SSH 접속)
새 버전을 올리기 전, 기존에 돌고 있던 낡은 박스들을 치워야 포트 충돌이 안 납니다.

```Bash
# 1. 실행 중인 컨테이너 멈추고 삭제 (f는 force, 강제삭제)
sudo docker rm -f backend frontend

# 2. (선택사항) 안 쓰는 이미지들을 지워 용량 확보
sudo docker image prune -a
```

Step 3. 서버에서 새 버전 실행하기 (Run)
가장 중요한 설정값들이 들어가는 단계입니다.

① MariaDB (데이터베이스)
DB는 한 번 켜두면 설정을 바꿀 때 빼고는 건드릴 일이 거의 없습니다.

```Bash
sudo docker run -d \
  --name mariadb \
  --network diary-net \
  -p 3306:3306 \
  -e MARIADB_ROOT_PASSWORD=1234 \
  -e MARIADB_DATABASE=diary \
  mariadb:latest
```
--network diary-net: 백엔드가 IP 대신 mariadb라는 이름으로 접속할 수 있게 같은 망에 넣음.

② Backend (Spring Boot)
```Bash
sudo docker run -d \
  -p 8080:8080 \
  --name backend \
  --network diary-net \
  -e SPRING_DATASOURCE_URL=jdbc:mariadb://mariadb:3306/diary \
  -e SPRING_DATASOURCE_USERNAME=root \
  -e SPRING_DATASOURCE_PASSWORD=1234 \
  asia-northeast3-docker.pkg.dev/haroo-799a8/diary-repo/backend-app
```
-e: 환경 변수 설정. 코드 수정 없이 DB 주소나 비번을 바꿀 수 있음.

③ Frontend (React/Vite)
```Bash
sudo docker run -d \
  -p 80:80 \
  --name frontend \
  --network diary-net \
  asia-northeast3-docker.pkg.dev/haroo-799a8/diary-repo/frontend-app
```
-p 80:80: 외부에서 http://도메인만 쳐도 들어오도록 웹 기본 포트(80)를 연결.

Step 4. 잘 돌아가는지 감시하기 (Monitoring)
명령어를 쳤는데 화면이 안 나온다면 아래 명령어로 원인을 찾습니다.

```Bash
# 1. 전체 박스 상태 보기 (STATUS가 Up 인지 확인)
sudo docker ps

# 2. 백엔드 로그 실시간 감시 (CORS 에러나 DB 에러 잡을 때 필수)
# -f: 로그가 올라올 때마다 실시간으로 화면에 출력
sudo docker logs -f backend

# 3. 컨테이너 내부의 파일이 궁금할 때 (예: DB 확인)
sudo docker exec -it mariadb mariadb -u root -p
```

💡 명령어 요약 활용법
코드 한 줄 고쳤을 때: build -> push -> SSH 접속 -> rm -> pull -> run

사이트가 안 열릴 때: sudo docker ps (박스 확인) -> sudo docker logs backend (에러 확인)

포트가 이미 사용 중일 때: netstat으로 PID 찾기 -> taskkill로 종료

5. 겪었던 주요 오류 및 해결책
❌ SqlExceptionHelper: Socket fail to connect
원인: 백엔드가 MariaDB 주소를 찾지 못함. (IP 주소가 바뀌거나 네트워크가 연결 안 됨)

해결: Docker Network(diary-net)를 만들고, 백엔드 설정에서 DB 주소를 IP 대신 컨테이너 이름인 mariadb로 지정함.

❌ Access-Control-Allow-Origin (CORS 에러)
원인: 브라우저 보안 정책상, 프론트(5173)와 백엔드(8080) 주소가 달라 요청을 거부함.

해결: 백엔드 SecurityConfig에서 AllowedOrigins에 도메인 주소와 IP를 명시하거나, 테스트 시 *(전체 허용)로 설정함.

❌ Port 8080 was already in use
원인: 이전에 실행했던 서버가 종료되지 않고 포트를 점유 중임.

해결: netstat -ano | findstr :8080으로 PID를 찾고 taskkill /f /pid [번호] 명령어로 강제 종료.

❌ Conflict. The container name is already in use
원인: 같은 이름의 도커 컨테이너가 이미 존재함.

해결: sudo docker rm -f [이름] 명령어로 기존 컨테이너를 삭제 후 재실행.

6. 운영 팁
수정 사항 반영: 로컬 수정 -> 빌드/푸시 -> GCP에서 rm -f로 삭제 -> pull -> 다시 run.

로그 확인: sudo docker logs -f backend (실시간 에러 확인 시 필수).
